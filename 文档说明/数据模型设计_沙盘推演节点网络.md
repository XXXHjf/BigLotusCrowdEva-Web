# 沙盘推演节点网络数据设计（静态 JSON 版）

## 1. 目标
用于“沙盘推演”页面里的“节点网络”图。

这个图的核心含义是：
1. 结点表示现实路线上的关键点位，例如入口、通道、场内区域、出口。
2. 边是有向边，方向表示人群移动方向。
3. 当对某个结点做反事实干预后，系统展示其他结点的人流量变化。

当前阶段先用静态 JSON 承载结果，不接数据库，不接后端接口。

---

## 2. 设计原则
- 只保留前端当前展示需要的最少字段。
- 结构尽量稳定，方便后续把师兄算法结果做一次转换后直接接入。
- 基础网络和推演结果分开，避免每次替换都复制整张图。

---

## 3. 文件位置（前端）
固定目录建议：`public/data/simulation/`

最小方案使用 2 个文件：
1. `node-network.json`
2. `counterfactual-results.json`

---

## 4. 两个文件分别负责什么

### 4.1 `node-network.json`
作用：
- 存放基线状态下的节点网络。
- 决定“节点网络”图初始显示的结点、边和基础人流量。

### 4.2 `counterfactual-results.json`
作用：
- 存放“对某个结点做某种干预”后的静态推演结果。
- 前端根据选中的结点和限流档位，找到对应结果并覆盖到图上。

---

## 5. 详细字段说明

## 5.1 `node-network.json`

### 标准结构
```json
{
  "nodes": [
    {
      "id": "node_gate_north",
      "label": "北入口",
      "value": 820
    },
    {
      "id": "node_corridor_a",
      "label": "A 通道",
      "value": 640
    }
  ],
  "edges": [
    {
      "source": "node_gate_north",
      "target": "node_corridor_a",
      "value": 0.82
    }
  ]
}
```

### 字段含义
顶层字段：
- `nodes`：结点数组。
- `edges`：有向边数组。

`nodes[*]` 子字段：
- `id`（必填，字符串）：结点唯一标识。
- `label`（必填，字符串）：结点显示名称。
- `value`（必填，数字）：当前人流量。

`edges[*]` 子字段：
- `source`（必填，字符串）：起点结点 ID。
- `target`（必填，字符串）：终点结点 ID。
- `value`（必填，数字）：边强度，可理解为通行强度、影响系数或流量强弱。

### 必须满足
1. `nodes[*].id` 不能重复。
2. `edges[*].source/target` 必须都能在 `nodes` 里找到。
3. 不允许 `source === target`。
4. 边方向必须与现实人群移动方向一致。
5. `nodes[*].value` 和 `edges[*].value` 必须是数字。

---

## 5.2 `counterfactual-results.json`

### 标准结构
```json
{
  "results": [
    {
      "nodeId": "node_corridor_a",
      "limitPercent": 50,
      "summary": "对 A 通道限流至 50% 后，下游区域通行能力下降，出口区排队长度上升。",
      "nodeUpdates": [
        {
          "nodeId": "node_corridor_a",
          "value": 320
        },
        {
          "nodeId": "node_exit_east",
          "value": 410
        }
      ],
      "edgeUpdates": [
        {
          "source": "node_corridor_a",
          "target": "node_exit_east",
          "style": "dashed"
        }
      ]
    }
  ]
}
```

### 字段含义
顶层字段：
- `results`：推演结果数组。

`results[*]` 子字段：
- `nodeId`（必填，字符串）：被干预的结点 ID。
- `limitPercent`（必填，数字）：干预档位。
- `summary`（必填，字符串）：该次推演的文字说明。
- `nodeUpdates`（必填，数组）：需要覆盖的人流量结果。
- `edgeUpdates`（可选，数组）：需要覆盖的边样式结果。

`nodeUpdates[*]` 子字段：
- `nodeId`（必填，字符串）：需要更新的结点 ID。
- `value`（必填，数字）：更新后人流量。

`edgeUpdates[*]` 子字段：
- `source`（必填，字符串）：边起点。
- `target`（必填，字符串）：边终点。
- `style`（必填，字符串）：当前前端最小只需要 `solid` 或 `dashed`。

### `limitPercent` 当前允许值
由于当前页面操作面板已固定为档位选择，因此静态结果也建议固定这 6 档：
- `0`
- `50`
- `80`
- `120`
- `150`
- `200`

说明：
- `0` 表示关闭通道。
- 其他值表示限流比例。

### 必须满足
1. `results[*].nodeId` 必须存在于 `node-network.json` 的 `nodes` 中。
2. 同一个 `(nodeId, limitPercent)` 只能出现一条结果。
3. `nodeUpdates[*].nodeId` 必须存在于 `node-network.json` 的 `nodes` 中。
4. `edgeUpdates[*].source/target` 必须能在 `node-network.json` 的 `edges` 中找到对应边。

---

## 6. 前端使用方式

### 6.1 初始显示
页面先读取 `node-network.json`：
- `nodes` 用于画结点。
- `edges` 用于画有向边。

### 6.2 进行推演
当用户选择：
- 某个结点 `nodeId`
- 某个档位 `limitPercent`

前端在 `counterfactual-results.json` 中找到对应结果，然后：
1. 用 `nodeUpdates` 覆盖相关结点的人流量。
2. 用 `edgeUpdates` 覆盖相关边的样式。
3. 用 `summary` 展示右侧“影响评估”说明。

补充说明：
- 当前页面已经正式接入这两份静态数据，不再依赖页面内即时 mock 计算。
- `node-network.json` 中的边在页面上按有向边渲染，箭头方向即人群移动方向。

---

## 7. 为什么这样拆
这样拆分的原因是：
1. 基础网络通常比推演结果更稳定。
2. 后续如果师兄给的是“每次干预一份结果”，只需要转成 `results` 数组即可。
3. 前端只做“基线 + 局部覆盖”，结构简单，不容易返工。

---

## 8. 最小替换规则
如果后面要用师兄的真实结果替换，只需要保证：
1. 基础网络能整理成 `nodes + edges`。
2. 每次反事实推演结果能整理成 `(nodeId, limitPercent)` 对应的一条记录。
3. 只提供变化过的结点放进 `nodeUpdates` 即可，不需要整图重复写一遍。

---

## 9. 常见错误
1. 边方向写反，导致人流路线语义错误。
2. `nodeId` 在结果表里存在，但在基础网络里不存在。
3. 同一个结点同一个档位出现多条结果。
4. `0` 档和“关闭通道”没有统一口径。
5. 为了方便导出，重复保存整张网络快照，导致文件冗余过大。

---

## 10. 最简结论
沙盘推演节点网络最小静态数据方案只需要两份数据：
1. 一份基础有向图：`node-network.json`
2. 一份反事实结果表：`counterfactual-results.json`

前者负责“原始网络长什么样”，后者负责“干预后哪些点和边变了”。  
这已经足够支撑当前页面的静态展示与后续真实数据替换。
