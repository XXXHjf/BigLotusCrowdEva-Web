# 模式解析页面模型数据设计

## 0. 这份文档是干什么的
这份文档告诉你：
1. 你需要给前端提供哪 4 个 JSON 文件。
2. 每个文件里每个字段是什么意思。
3. 哪些字段必须有，哪些可以没有。
4. 你导出后怎么快速自查，避免前端页面显示异常。

你不需要懂前端工程，只需要按本说明导出数据。

---

## 1. 总体思路（先理解）
我们把页面拆成 3 块：

1. 左侧地图：显示所有聚类结点（点位）
2. 中间图：点击某个结点后，显示与它相连的有向关系（source -> target）
3. 右侧四卡：显示这个结点是否属于四种模式（因果链/枢纽/社区/关键节点）

因此数据也拆成 4 个文件，分别负责不同事情。

---

## 2. 文件放置位置（前端）
固定目录：`public/data/pattern-analysis/`

必须存在这 4 个文件：
1. `nodes.json`
2. `edges.json`
3. `mode-membership.json`
4. `mode-graphs.json`

简化约定：
- 顶层不要求 `version`、`generatedAt`、`source`。
- 这三个字段统一不作为必填，师兄可以不提供。

---

## 3. 四个文件分别负责什么

### 3.1 `nodes.json`（结点总表）
作用：
- 存放系统里“所有聚类结点”的基础信息。
- 左侧地图点位完全由它决定。

### 3.2 `edges.json`（关系总表）
作用：
- 存放所有有向边（source -> target）。
- 中间图点击结点后，会在这个表里筛选“与该结点有关的边”。

### 3.3 `mode-membership.json`（结点属于哪些模式）
作用：
- 告诉前端某个结点命中了哪些模式（0~4 种）。
- 右侧四卡是否显示图，先看这个文件。

### 3.4 `mode-graphs.json`（四种模式的子图定义）
作用：
- 定义每种模式自己的子图（节点+边）。
- 当右侧某个模式“命中”时，实际渲染的数据来自这里。

### 3.5 四种模式的强语义约束（必须遵守）
以下外形定义以 [四种模式子图的说明](/mnt/c/Users/hjf20/Documents/Codings/ReactPrograms/BigLotusCrowdEva/文档说明/四种模式子图的说明.md) 为准，本节仅做工程化映射。

1. 因果链 `causalChain`
- 以被选中结点为起点（起因点）。
- 子图必须是单向链式结构。
- 按时间或因果顺序依次连接。
- 最多展示 3 层（即最多 3 个结点）。

2. 枢纽节点 `hubNode`
- 以被选中结点为中心。
- 子图必须是有向星型结构。
- 表达“中心结点指向多个结点”的辐射关系。

3. 社区结构 `community`
- 以被选中结点为切入点。
- 子图应为密集网状/团块状。
- 群内多向连线明显，内部连接紧密，外部连接稀疏。

4. 关键结点 `keyNode`
- 以被选中结点为桥梁。
- 子图应体现桥接型结构（连接原本不互通区域）。
- 重点体现其在全局连通与流转控制中的“咽喉”作用。

## 4. 详细字段说明

## 4.1 `nodes.json`

### 标准结构
```json
{
  "nodes": [
    {
      "id": "cluster_502",
      "lng": 120.2235242857,
      "lat": 30.2305188571,
      "label": "cluster_502",
      "tags": ["hub"],
      "extra": {
        "其他可能的字段": "字段值",
        "type": "hub",
        "influenceCount": 9,
        "score": 0.3499,
        "rank": 1
      }
    }
  ]
}
```

### 字段含义
- `nodes`：结点数组

`nodes[*]` 子字段：
- `id`（必填，字符串）：结点唯一标识，建议格式 `cluster_数字`
- `lng`（必填，数字）：经度
- `lat`（必填，数字）：纬度
- `label`（可选，字符串）：显示名称，不填可等于 `id`
- `tags`（可选，字符串数组）：算法角色标签，例如 `hub`、`chain_node`
- `extra`（可选，对象）：保留算法原始字段，前端默认不依赖，但建议保留用于追溯

### 必须满足
1. 同一文件内 `id` 不能重复。
2. `lng/lat` 必须是数字，不能是字符串。
3. 不允许 `null` 坐标。

---

## 4.2 `edges.json`

### 标准结构
```json
{
  "edges": [
    {
      "source": "cluster_502",
      "target": "cluster_169",
      "weight": 12,
      "relationType": "flow",
      "modes": ["hubNode"],
      "extra": {
        "其他可能的字段": "字段值"
      }
    }
  ]
}
```

### 字段含义
- `edges`：边数组

`edges[*]` 子字段：
- `source`（必填，字符串）：起点结点 ID
- `target`（必填，字符串）：终点结点 ID
- `weight`（可选，数字）：边权重/流量，不填默认按 1
- `relationType`（可选，字符串）：关系类型，建议固定 `flow`
- `modes`（可选，字符串数组）：该边属于哪些模式（用于分析，可不填）
- `extra`（可选，对象）：保留算法原始边字段

### 必须满足
1. `source` 和 `target` 必须都在 `nodes.json` 里存在。
2. 不允许 `source === target`（自环边）。
3. 同方向重复边要去重（同一 `source->target` 保留一条）。

---

## 4.3 `mode-membership.json`

### 标准结构
```json
{
  "memberships": [
    {
      "nodeId": "cluster_502",
      "modes": ["hubNode"],
      "isAnchor": true,
      "extra": {
        "其他可能的字段": "字段值"
      }
    },
    {
      "nodeId": "cluster_169",
      "modes": [],
      "isAnchor": false
    }
  ]
}
```

### 字段含义
`memberships[*]` 子字段：
- `nodeId`（必填，字符串）：结点 ID
- `modes`（必填，数组）：该结点命中的模式列表，可为空数组
- `isAnchor`（必填，布尔）：是否为任一模式锚点
- `extra`（可选，对象）：保留该节点的模式附加信息（置信度、分数来源等）

### `modes` 允许值
- `causalChain`
- `hubNode`
- `community`
- `keyNode`

### 必须满足
1. `nodeId` 必须存在于 `nodes.json`。
2. 一个 `nodeId` 只能出现一次。
3. `isAnchor` 建议与 `mode-graphs.json` 的 `anchors` 一致。

### 师兄原始数据转正式格式时的命名映射
如果师兄原始输出不是当前前端命名，可按下表统一转换：
- `causal_chain` -> `causalChain`
- `hub` / `hub_node` -> `hubNode`
- `community` -> `community`
- `key_node` / `critical_node` -> `keyNode`

---

## 4.4 `mode-graphs.json`

### 标准结构
```json
{
  "modeGraphs": {
    "causalChain": {
      "description": "因果链",
      "anchors": ["cluster_604", "cluster_198"],
      "extra": {
        "其他可能的字段": "字段值",
        "chain_id": 23,
        "steps": [
          { "step": 1, "pointId": "cluster_604" },
          { "step": 2, "pointId": "cluster_198" }
        ]
      },
      "graph": {
        "nodes": [
          { "id": "cluster_604", "role": "anchor", "extra": { "其他可能的字段": "字段值", "type": "chain_node" } },
          { "id": "cluster_198", "role": "anchor", "extra": { "type": "chain_node" } },
          { "id": "cluster_469", "role": "member", "extra": { "type": "surrounding_context" } }
        ],
        "edges": [
          { "source": "cluster_604", "target": "cluster_198", "weight": 10, "extra": { "其他可能的字段": "字段值" } },
          { "source": "cluster_198", "target": "cluster_469", "weight": 7, "extra": {} }
        ]
      }
    }
  }
}
```

### 字段含义
- `modeGraphs`：四种模式对象（可以缺一部分）
- `modeGraphs.<mode>.description`：模式名称
- `modeGraphs.<mode>.anchors`：该模式锚点列表
- `modeGraphs.<mode>.extra`（可选）：保留该模式特有原始字段（如 `chain_id`、`steps`）
- `modeGraphs.<mode>.graph.nodes`：模式子图里的节点
- `modeGraphs.<mode>.graph.edges`：模式子图里的边

`graph.nodes[*]` 子字段：
- `id`（必填）：结点 ID
- `role`（可选）：模式内角色，常用值：`anchor`、`member`、`bridge`、`context`
- `extra`（可选）：保留节点原始字段

`graph.edges[*]` 子字段：
- `source`（必填）
- `target`（必填）
- `weight`（可选）
- `extra`（可选）：保留边原始字段

### 必须满足
1. `anchors` 里的 ID 必须存在于 `graph.nodes`。
2. `graph.edges` 引用的结点必须存在于 `graph.nodes`。
3. 建议 `graph.nodes` 里的 ID 同时存在于 `nodes.json`。

---

## 5. 四个文件之间的关系

把它当成 4 张“静态表”：
1. `nodes.json`：节点主表
2. `edges.json`：边主表（外键指向节点）
3. `mode-membership.json`：节点-模式关系表
4. `mode-graphs.json`：模式子图快照表

前端读取逻辑：
1. 先加载四个文件。
2. 点击结点时：
- 中间图 = `edges.json` 过滤出 `source==id 或 target==id`。
- 右侧卡 = 先看 `mode-membership.json` 的 `modes`，命中的模式再去 `mode-graphs.json` 取子图。

---

## 6. 常见错误（请优先避免）
1. `id` 拼写不统一（`cluster-1` 和 `cluster_1` 混用）。
2. 坐标导成字符串（`"120.22"`）。
3. 边引用了不存在的结点。
4. `mode-membership` 漏了某些结点，导致颜色/模式判定异常。
5. `anchors` 与 `isAnchor` 不一致。
6. 沿用师兄原始模式名但未做命名映射，导致页面右侧模式子图无法命中。
